## 1. 목적

시스템 구성 요소 간 역할과 책임을 명확히 정의하고, 상호작용 방식을 표준화합니다.

---

## 2. 주요 에이전트

### UI Agent

**역할**

* 상태를 렌더링
* 사용자 이벤트 전달

**규칙**

* 비즈니스 로직 금지
* 플레이어 직접 제어 금지

---

### ViewModel Agent

**역할**

* UI 상태 관리
* 이벤트 → 상태 변환 처리

**규칙**

* 단일 상태 소스 유지
* Flow 기반 상태 전달

---

### Player Agent (PlayerController)

**역할**

* 재생 제어
* 플레이어 라이프사이클 관리

**규칙**

* ExoPlayer 추상화
* UI 의존성 제거

---

### Data Agent

**역할**

* 영상 데이터 제공

**규칙**

* 재생 로직 포함 금지

---

## 3. 상호작용 흐름

1. UI에서 스크롤 이벤트 감지
2. ViewModel로 이벤트 전달
3. ViewModel이 활성 인덱스 갱신
4. PlayerController에 재생 요청
5. 플레이어 상태 반영

---

## 4. 설계 원칙

* 관심사 분리
* 예측 가능한 상태 흐름
* 라이프사이클 안정성
* 성능 중심 설계

---

## 5. 제약 조건

* 초기 설계: 단일 플레이어 사용
* 최소 메모리 사용
* 빠른 초기 로딩

---

## 6. 확장 전략

* PlayerController → 풀 구조로 확장
* 캐싱 레이어 추가
* 도메인 로직 확장

---

## 7. 주요 트레이드오프

| 결정         | 장점     | 비용       |
| ---------- | ------ | -------- |
| 단일 플레이어    | 메모리 절약 | 재바인딩 복잡도 |
| 프리로딩       | UX 개선  | 리소스 증가   |
| LazyColumn | 구현 단순  | 제어 한계    |

---

## 8. 요약

이 시스템은 기능 확장성보다 **안정성, 성능, 상태 흐름의 명확성**을 우선으로 설계되었습니다.
